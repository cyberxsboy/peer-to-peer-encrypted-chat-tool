# 去中心化加密P2P聊天工具开发文档

**项目名称**：SecureP2P Chat  
**项目类型**：去中心化端到端加密点对点聊天应用  
**主要技术栈**：Node.js、Electron、React、libp2p、PostgreSQL  
**核心特性**：文件传输、群聊、媒体预览、语音转文字、阅后即焚、端到端加密  
**协议**：Noise Protocol（Noise_IK_25519_AESGCM_SHA256）  
**身份认证**：中心身份服务器 + 本地私钥保护  

---

## 第一章 项目概述

### 1.1 项目背景

随着互联网技术的快速发展和用户隐私意识的不断提升，传统的中心化即时通讯工具面临着数据泄露、隐私侵犯和审查风险等多重挑战。本项目旨在构建一个**真正的去中心化端到端加密P2P聊天工具**，让用户能够在一个安全、私密的环境中自由交流，不再受限于中心化服务器的监控和数据存储。

本项目采用Node.js作为核心技术栈，利用libp2p实现真正的点对点通信，使用Noise协议提供前向保密的端到端加密，并结合中心身份服务器实现用户身份的可发现性。这种架构既保证了通信的隐私性和安全性，又保留了用户体验的便利性。

### 1.2 设计原则

**真正的点对点（True Peer-to-Peer）**

本项目坚持真正的去中心化架构，不在中央服务器上存储任何消息内容或社交图谱信息。消息的传输完全通过点对点加密通道进行，只有通信的双方（或群组成员）能够解密和查看消息内容。中央身份服务器仅负责用户身份的验证和公钥的分发，不参与任何消息的中转或存储。

**端到端加密（End-to-End Encryption）**

所有通信内容（文本、文件、媒体、语音、链接预览、邀请等）在发送方设备上就已经被加密，只有接收方能够解密。即使是中央身份服务器被攻破，攻击者也无法获取任何通信内容。这种端到端加密机制确保了用户的通信隐私得到最大程度的保护。

**去中心化身份（Decentralized Identity）**

用户的身份由加密密钥对定义，用户名和邮箱仅作为可被发现和查找的别名，由中心身份服务器进行管理。这种设计使得用户的真实身份与其加密身份分离，提供了更好的隐私保护。即使中心身份服务器被攻破，攻击者也无法直接将用户名与真实身份关联。

**最小信任（Minimal Trust）**

即使中央身份服务器被攻破，也不会泄露用户的私钥或消息内容。服务器仅存储经过用户登录密码派生密钥加密后的公钥，无法直接获取明文公钥或私钥。这种最小信任设计确保了系统在中心服务器被攻破的情况下仍能保护用户隐私。

**渐进式交付（Progressive Delivery）**

项目采用渐进式开发策略，首先实现核心的安全消息传输功能，然后逐步添加多媒体传输、群组管理、阅后即焚等高级功能。这种策略确保每个阶段都能产出可用的成果，便于及时获取反馈和调整方向。

### 2.2 技术架构详解

**libp2p - 去中心化P2P网络**

libp2p是本项目的网络核心，负责节点发现、连接建立和数据传输。它支持多种传输协议，包括TCP、WebSocket、WebRTC和mDNS，能够适应不同的网络环境。libp2p还支持多路复用，可以在单一连接上同时传输多种数据流。

**Noise协议 - 端到端加密**

Noise协议是本项目的加密核心，采用了Noise_IK_25519_AESGCM_SHA256套件。这个套件使用Curve25519进行密钥交换，使用AES-256-GCM进行对称加密，并使用SHA-256进行消息认证。Noise IK模式提供了前向保密和双方身份验证。

**Argon2id - 密码哈希**

Argon2id是目前最强的密码哈希算法之一，能够有效抵抗GPU和ASIC攻击。本项目使用Argon2id对用户密码进行哈希存储，并使用相同的算法派生加密密钥来保护本地私钥。

**Prisma + PostgreSQL - 数据存储**

PostgreSQL是一个成熟、稳定的关系型数据库，适合存储用户账户和公钥等结构化数据。Prisma提供了类型安全的数据库操作API，简化了数据库操作的复杂度。

## 第三章 身份服务器设计

### 3.1 服务器职责

身份服务器是本系统的可选组件，主要负责以下职责：

**用户账户管理**

- 用户名和邮箱的唯一性验证
- 用户密码的安全存储（Argon2id哈希）
- 用户注册、登录、登出流程
- 密码重置功能

**公钥管理**

- 存储用户加密后的公钥
- 提供公钥查询接口
- 支持公钥更新和轮换

**会话管理**

- JWT令牌的颁发和刷新
- 会话状态维护
- 会话吊销

### 3.2 数据库设计

#### 3.2.1 用户表（User）

```prisma
model User {
  id                String      @id @default(uuid())
  username          String      @unique
  email             String      @unique
  passwordHash      String      // Argon2id哈希（包含盐）
  salt              Bytes      @db.Binary(16)  // Argon2id盐
  createdAt         DateTime    @default(now())
  updatedAt         DateTime    @updatedAt
  isActive          Boolean     @default(true)
  failedLoginCount  Int         @default(0)   // 失败登录次数
  lockedUntil      DateTime?                     // 锁定截止时间
  
  publicKey        PublicKey?  @relation("UserPublicKey")
  sessions        Session[]
  passwordResets   PasswordReset[]
}
```

**字段说明**：

- `id`：用户唯一标识，UUID格式
- `username`：用户名，3-20字符，仅支持字母、数字、下划线
- `email`：邮箱地址，唯一
- `passwordHash`：Argon2id哈希值，包含参数和盐
- `salt`：用于Argon2id的盐，16字节
- `isActive`：账户是否激活
- `failedLoginCount`：连续失败登录次数，用于安全防护
- `lockedUntil`：账户锁定截止时间

#### 3.2.2 公钥表（PublicKey）

```prisma
model PublicKey {
  userId        String    @unique @map("user_id")
  user          User      @relation("UserPublicKey", fields: [userId], references: [id], onDelete: Cascade)
  pubKeyEnc     Bytes               // XChaCha20-Poly1305加密的公钥
  pubKeyHash    Bytes    @db.Binary(32)  // SHA-256哈希（非敏感）
  updatedAt    DateTime  @default(now())
  
  @@index([pubKeyHash])
}
```

**字段说明**：

- `userId`：关联的用户ID
- `pubKeyEnc`：使用用户登录密码派生的密钥加密后的公钥
- `pubKeyHash`：公钥的SHA-256哈希，用于快速查找但不泄露内容
- `updatedAt`：最后更新时间

### 3.4 安全防护

**密码安全**

- 使用Argon2id进行密码哈希，参数：timeCost=2，memoryCost=64MB，parallelism=4
- 盐长度为16字节
- 服务器不存储明文密码

**暴力破解防护**

- 每个IP地址和账户每15分钟最多5次登录尝试
- 连续失败5次后锁定账户15分钟
- 返回统一的错误信息，防止用户名枚举

**传输安全**

- 强制HTTPS
- 启用HSTS
- 强制Content-Security-Policy

**数据保护**

- 公钥在服务器端使用用户密码派生的密钥加密
- 服务器无法获取明文公钥

## 第四章 libp2p节点设计

### 4.1 节点职责

libp2p节点是本系统的核心组件，运行在Electron主进程中，负责以下职责：

**身份管理**

- 生成和管理节点身份密钥对
- 私钥使用Argon2id派生的密钥加密存储
- 公钥用于节点身份识别

**网络连接**

- 通过Noise协议建立加密连接
- 支持多种传输协议（TCP、WebSocket、WebRTC、mDNS）
- 处理NAT穿透

**数据流管理**

- `/chat/1.0.0` - 加密消息流
- `/file/1.0.0` - 文件传输流
- `/voice/1.0.0` - 语音流

**对等发现**

- mDNS用于局域网发现
- 引导节点列表用于初始连接
- 可选的DHT用于广域网发现

### 4.2 Noise IK握手流程

Noise IK（Interactive Key）协议是一种前向保密的密钥交换协议，提供了互动的身份验证。下面是详细的握手流程：

**步骤1：握手初始化**

```
发起方A                    响应方B
   |                         |
   |---- e (临时公钥) ------>|
   |<--- e (临时公钥) -------|
```

- 发起方生成临时密钥对(e, E)
- 响应方生成临时密钥对(e, E)
- 双方交换临时公钥

**步骤2：DH拉链**

```
   DH1 = DH(响应方静态私钥, 发起方临时公钥)
   DH2 = DH(发起方静态私钥, 响应方临时公钥)
   (IK模式下，这是两次独立的DH运算)
   CK = HKDF(NIL, DH1 || DH2)
```

**步骤3：静态密钥交换**

```
   |---- s (加密的静态公钥) -->|
   |<-- s (加密的静态公钥) ----|
```

- 使用CK加密静态公钥
- 双方验证对方的身份

**步骤4：会话密钥派生**

```
   k_e = HKDF_expand(CK, "k_e", 32)  // 发送密钥
   k_d = HKDF_expand(CK, "k_d", 32)  // 接收密钥
   nonce = 0
```

### 4.3 消息格式设计

所有应用层消息在Noise加密之后传输，遵循以下统一格式：

```typescript
interface ChatMessage {
  msgId: string;              // UUID v4
  from: string;               // 发送方peerId (Base58)
  to: string;                // 接收方peerId或groupId (Base58)
  timestamp: number;          // UNIX毫秒时间戳
  type: MessageType;         // 消息类型
  payload: EncryptedPayload;// 加密后的载荷
  replyTo?: string;          // 被回复的消息ID
  burnAfterSec?: number;    // 阅后即焚秒数（null表示不焚毁）
}

enum MessageType {
  TEXT = 0,           // 文本消息
  FILE = 1,           // 文件消息
  VOICE = 2,          // 语音消息
  LINK_PREVIEW = 3,   // 链接预览
  INVITE = 4,          // 群组邀请
  RECEIPT = 5,         // 回执
  BURN_NOTICE = 6      // 焚毁通知
}
```

**消息加密流程**：

```
1. 序列化消息为JSON
2. (可选) 使用snappy压缩
3. 生成12字节随机nonce
4. 使用AES-256-GCM加密
5. 组装传输帧：[nonce][ciphertext][tag]
```

### 4.4 流协议定义

#### 4.4.1 聊天协议（/chat/1.0.0）

用于传输所有聊天消息，包括文本、文件、语音、链接预览等。

**消息帧格式**：

```
[1字节标志位][12字节nonce][密文][16字节认证标签]
```

标志位定义：

```
bit 0: 是否压缩
bit 1: 是否有replyTo
bit 2: 是否阅后即焚
bits 3-7: 保留
```

#### 4.4.2 文件协议（/file/1.0.0）

用于传输大文件，采用分块传输。

**分块格式**：

```
[4字节总块数][4字节当前块索引][数据块]
```

#### 4.4.3 语音协议（/voice/1.0.0）

用于实时语音消息录制和传输。

**语音帧格式**：

```
[2字节序列号][Opus编码数据]
```

### 4.5 对等发现机制

**mDNS（局域网发现）**

- 自动发现同一局域网内的其他节点
- 无需任何中央服务器
- 适用于局域网环境

**引导节点（Bootstrap）**

- 预配置的公开节点列表
- 用于初始连接建立
- 可以自建或使用社区节点

**DHT（分布式哈希表）**

- 可选的广域网发现
- 支持根据peerId查找节点
- 提供更好的可扩展性

- 通过加密邀请 blob 邀请

### 5.4 邀请流程

**生成邀请**：

```
1. 群主生成随机群组密钥GK
2. 使用邀请密码（如果有）派生的密钥加密GK
3. 如果没有密码，使用群主私钥直接加密GK
4. 签名整个邀请blob
5. 生成邀请链接或二维码
```

**接受邀请**：

```
1. 解析邀请blob
2. 验证邀请者签名
3. 如果有密码，输入密码并验证
4. 解密群组密钥GK
5. 保存群组信息和GK
6. 向群主发送加入确认
```

```

4. 显示消息内容

### 7.1 机制概述

阅后即焚是一种临时消息功能，消息在查看后或指定时间后自动销毁，无法恢复。

5. 发送加密消息

### 7.4 接收流程

```
1. 使用会话密钥解密k_burn
2. 使用k_burn解密消息载荷
3. 启动倒计时UI
4. 用户查看消息或倒计时结束
5. 零化k_burn和明文载荷
6. 显示焚毁动���

### 7.5 群组阅后即焚

```

// 用户名验证
const validateUsername = (value: string): ValidationResult => {
  if (!value) return { valid: false, message: '请输入用户名' };
  if (!/^[a-zA-Z0-9_]{3,20}$/.test(value)) {
    return { valid: false, message: '仅支持字母、数字、下划线，长度3-20' };
  }
  return { valid: true, message: '用户名可用' };
};

// 邮箱验证
const validateEmail = (value: string): ValidationResult => {
  if (!value) return { valid: false, message: '请输入邮箱' };
  if (!/^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(value)) {
    return { valid: false, message: '请输入有效的邮箱地址' };
  }
  return { valid: true, message: '邮箱格式正确' };
};

// 密码验证
const validatePassword = (value: string): ValidationResult => {
  if (!value) return { valid: false, message: '请输入密码' };
  if (value.length < 8) {
    return { valid: false, message: '密码至少8位' };
  }
  const hasUpper = /[A-Z]/.test(value);
  const hasLower = /[a-z]/.test(value);
  const hasNumber = /[0-9]/.test(value);
  const types = [hasUpper, hasLower, hasNumber].filter(Boolean).length;
  if (types < 2) {
    return { valid: false, message: '需包含大小写字母和数字中的至少两类' };
  }
  return { valid: true, message: '密码强度满足' };
};
```

## 第十一章 安全考虑

### 11.1 端到端加密

**密钥生成**：

- 使用密码学安全的随机数生成器
- Curve25519用于密钥交换
- AES-256-GCM用于对称加密

**前向保密**：

- 每次会话生成新的临时密钥
- 长期密钥不参与消息加密
- 定期轮换群组密钥

### 11.2 本地存储安全

**私钥保护**：

- 使用Argon2id派生的密钥加密
- 不以明文存储
- 使用后立即零化

**消息存储**：

- 本地消息也应加密存储
- 阅后即焚消息不留存
- 敏感数据使用安全擦除

### 11.3 网络安全

**传输加密**：

- 全部使用HTTPS（服务器）
- 全部使用Noise（节点间）

**防攻击**：

- 登录速率限制
- 防止暴力破解
- 防止重放攻击

### 11.4 隐私保护

**数据最小化**：

- 服务器不存储消息内容
- 不记录社交图谱
- 不追踪用户行为

**身份分离**：

- 用户名与peerId分离
- 公钥与身份信息分离
- 支持匿名使用
